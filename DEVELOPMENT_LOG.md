# 網站自動 QA 檢查器 — 開發日誌

## 階段一：需求討論與方案選型（2026-07-03）

### 對話過程

使用者提出：女友公司的美編做的網站每次都有 QA 問題——按鈕按下去沒反應、LINE 加好友按鈕連到「下載 LINE APP」而不是 QR Code 加好友頁等，希望有「自動 QA 網站的東西」，做給女友公司的業務自行檢查使用。

**討論出的關鍵需求（第二輪對話確立）：**
> 「業務把要檢查的網站貼網址去某個地方，就可以幫檢查，最後顯示網站要改的錯誤」

即：不是工程師用的 CLI，而是**業務自助操作的網頁工具**。

### 技術決策

1. **為何不用現成工具？** 死連結檢查器（linkinator）不會點按鈕、Lighthouse 只看效能、視覺回歸工具看不出按鈕沒綁事件。「LINE 加好友連結行為」這種台灣在地需求沒有現成工具涵蓋。
2. **架設位置**（使用者選擇）：**Vercel** — 使用者已有帳號，免費、免維護，業務開網址就能用。取捨：單次檢查上限 5 分鐘（`maxDuration: 300`），對美編做的一頁式/小型網站足夠；超大網站會回報「僅檢查前 N 頁」。
3. **瀏覽器引擎**：`puppeteer-core + @sparticuz/chromium`（而非 Playwright）— 這是 Vercel serverless 上跑無頭 Chrome 的最成熟組合，版本配對有官方支援。本機開發 fallback 到本機 Chrome/Edge。
4. **死按鈕偵測策略**：不用靜態分析（addEventListener 綁的事件從 DOM 看不到），改為**真實點擊 + 觀察反應**：點擊後 1.2 秒內偵測「跳轉／開新視窗／alert／發出網路請求／DOM 變化」，全部皆無 → 判定死按鈕。跳過表單內按鈕與 submit（避免真的送出表單）。
5. **進度回饋**：API 以 NDJSON streaming 回傳，前端即時顯示「載入頁面…／檢查 N 個連結…」等進度。
6. **報告分享**：不建資料庫，前端一鍵產出獨立 HTML 報告檔（截圖以 base64 內嵌），業務直接傳檔案給美編。
7. **Repo 慣例**：public repo，README 含需求起因，本日誌記錄對話與決策，程式只存 GitHub。

### LINE 連結檢查規則（本工具核心特色）

| 情況 | 判定 |
|---|---|
| `lin.ee/xxx`、`line.me/R/ti/p/@id` | ✅ 正確格式，另驗證連結可開啟、未跳轉到下載頁 |
| `line.me/*/download` | 🔴 連到 APP 下載頁 |
| `line://` 深層連結 | 🔴 電腦版點了沒反應 |
| APP 商店的 LINE 頁面 | 🔴 連到商店下載頁 |
| 其他 line.me 網址 | 🟡 請人工確認 |

## 階段二：實作與測試除錯（2026-07-03）

以「故意做錯的測試網頁」驗證抓錯能力（7 個故意錯誤 + 2 個正常對照組），過程中修正三個問題：

1. **誤判正常按鈕為死按鈕**：按鈕的 onclick 改了文字內容，但 MutationObserver 沒監聽 `characterData`，且反應門檻設 `>2` 太高 → 補上 characterData 監聽、門檻改為 `>0`。
2. **漏抓 `href="#"` 假連結**：點擊後網址從 `/` 變 `/#` 被誤判為「有跳轉」→ 比較網址時先去掉 hash。
3. **漏抓手機版跑版**：手機模擬下 Chrome 有 shrink-to-fit 行為，會把 layout viewport 撐大到內容寬度（1200px 內容 → `window.innerWidth` 也變 1208），`scrollWidth > innerWidth` 永遠不成立 → 改為直接與手機實際寬度 390px 比較。另補「缺少 viewport meta」檢查。

最終驗證：7 個故意錯誤全數抓到，正常按鈕與正常連結無誤判。

**測試頁錯誤覆蓋**：死按鈕、`href="#"`、LINE 下載頁連結、`line://` 連結、404 連結、破圖、手機跑版、console 錯誤。

## 階段三：上線、業務使用流程確認與示範頁（2026-07-03）

### 對話過程

1. 部署至 Vercel 後，使用者確認核心使用情境：「**要把檢測給業務用**——業務把美編的網頁自己貼去檢查，檢測完業務要能看懂要改什麼、再通知美編」。
2. 確認了兩件事：
   - **分發方式**：介面就是一個公開網址（https://website-qa-checker.vercel.app），LINE 群組貼一次給業務即可，免安裝免登入。
   - **業務→美編的溝通**：報告全部用白話文（「按鈕『立即購買』按下去沒有任何反應」），分「🔴 必修錯誤／🟡 建議修正」兩級，業務不需判斷嚴重性；按「下載報告（傳給美編）」取得含截圖的 HTML 檔，直接丟 LINE/Email 給美編。
3. 新增 **示範用故障網頁 `/demo.html`**（內含 7 種故意錯誤：死按鈕、`href="#"`、LINE 下載頁連結、`line://` 連結、404、破圖、手機跑版），供業務培訓時練習跑一次、認識報告長相。
4. 以 Puppeteer 模擬業務完整操作（開頁→貼網址→檢查→出報告）截圖驗證，介面與報告呈現正常。

> 美編說做好了 → 業務貼網址檢查 → 有紅字就下載報告傳給美編 → 改完再查一次 → 全過才驗收

## 階段四：以 5 個真實美編網站實測、優化抓錯規則（2026-07-06）

### 對話過程

使用者提供 5 個美編製作（已校正過）的網站作為實測參考：4 個 Weebly（嘟嘟車泰奶、永成麵包、鑫花溪、雞雞叫吐司）+ 1 個 Framer（張美阿嬤）。以正式站「連同子頁面」模式掃描。

### 實測結果與發現

**真實錯誤（工具立功）：**
- 永成麵包：一張圖片的 src 被填成網頁連結（`...menu.html`）導致破圖 → 紅色錯誤，正確。

**誤判與雜訊（工具要改）：**
1. **「(無文字)死按鈕」爆量誤判（16 筆）**：實際檢查後發現是 Weebly 把圖片包在 `<a>` 連結框但未設連結——多數是無需點擊的裝飾圖，不該報紅色錯誤，且「(無文字)」讓人不知道是哪一顆。
2. **console 雜訊**：CSP report-only、Permissions policy 等平台無害訊息被當錯誤；Framer 的 React hydration 訊息每頁重複報一次（5 頁 5 筆）。
3. **外送平台 403**：UberEats/foodpanda 一律阻擋機器人，措辭讓業務誤以為連結壞掉。

### 解決方案（commit `9f693dc`）

1. 無文字按鈕改以「圖片檔名/alt/class」描述；Weebly「圖片連結框未設連結」獨立為**黃色警告**新類別「未設連結的圖片」，附說明讓業務判斷該圖是否本該可點。
2. console 錯誤加 `IGNORE_CONSOLE` 無害雜訊過濾 + 跨頁去重（同錯誤全站只報一次）。
3. 外送/訂位平台 403 改為安撫性措辭：「平台會阻擋自動檢查，連結本身通常是正常的」。

## 階段五：使用者回報誤判——泰速竹北店案例（2026-07-06）

### 問題回報

使用者實測 `thaispeed-zhubei.framer.website` 後回報兩個問題：
1. 報告說按鈕「Previous」按下去沒反應，但網站上根本看不到這顆按鈕。
2. UberEats 連結被列為失效，但人工點開是正常的。

### 除錯過程與根因

1. **「Previous」**：實際檢查 about-us 頁 DOM，發現是 Framer 相簿輪播的「上一張」箭頭。輪播停在第一張時 Framer 將它設為 `opacity: 0` + `pointer-events: none`——對使用者是隱形且點不到的，但檢查器的可見性判斷只看 offsetWidth/Height（元素有 40×40 尺寸），於是照點並誤報死按鈕。
2. **UberEats**：外送平台一律以 403 阻擋機器人（先前已軟化措辭），但只要出現在警告列表就會讓業務誤以為連結壞掉。

### 解決方案（可見性判斷升級）

1. 點擊測試前加 `getComputedStyle` 檢查：`opacity < 0.05`、`visibility: hidden`、`pointer-events: none` 的元素視為「使用者看不到」直接跳過。
2. 輪播/滑動元件的前後箭頭（aria-label 為 Previous/Next，或 class 含 carousel/slider/swiper/slick 等）在邊界本來就沒反應，屬正常行為，一律不測。
3. 已知外送/訂位平台（UberEats/foodpanda 等）的 401/403 **完全不列入報告**（真正壞掉的連結會回 404，仍會抓到）。
4. Framer/React 的 hydration 類 console 訊息（Minified React error #405/#418/#423/#425）加入無害雜訊過濾——每個 Framer 網站都有，業務無法採取行動。

### 驗證結果

- 泰速竹北店重掃：🔴 0 錯誤、🟡 0 警告（原誤判全部消失）
- demo.html 迴歸測試：6 個真實錯誤全數照抓，無漏抓

## 階段六：定位調整——移除技術性檢查，聚焦一般使用情境（2026-07-08）

使用者確立工具定位：**面向沒有程式能力的使用者，只檢查一般使用情況會遇到的問題**。

- 移除「程式錯誤」（console/JS 錯誤）與「資源載入失敗」兩類檢查——業務看不懂也無法採取行動，且實測中多為平台雜訊。
- 使用者真正會遇到的載入問題仍有涵蓋：破圖（獨立檢查）、頁面打不開、連結失效。
- 另一調整（2026-07-07）：預設檢查深度改為「連同站內子頁面」。

保留類別：死按鈕、LINE連結、失效連結、破圖、未設連結的圖片、手機版、頁面載入、基本檢查（標題）。
